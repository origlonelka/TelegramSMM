# ТЗ: TelegramSMM v2 (безопасность, монетизация, админ-панель, постинг в рекламные чаты)

## 1) Цель
Превратить текущий бот в коммерческий сервис с платным доступом, безопасной архитектурой и расширенной админкой:
- исправить критичные уязвимости и эксплуатационные дефекты;
- добавить триал 24 часа и подписки через ЮKassa;
- внедрить полноценную админ-панель;
- добавить режим публикации в разрешенные рекламные чаты по строгим лимитам.

## 2) Обязательные исправления (из ревью)

### SEC-01: Безопасный импорт tdata (ZIP)
Требования:
- запрет absolute paths и `..` в путях архива;
- запрет symlink/hardlink entries;
- лимиты на размер: `max_total_unpacked_size`, `max_file_size`, `max_files`;
- разрешать только ожидаемую структуру (`tdata/...`);
- при нарушении: прерывать импорт, логировать причину, не писать файлы вне temp-директории.

Критерии приемки:
- архив с `../` отклоняется;
- архив с symlink отклоняется;
- архив, превышающий лимиты, отклоняется.

### SEC-03: Доступ в админку строго по Telegram ID (ID-only)
Требования:
- проверка прав только по `from_user.id`;
- убрать `ADMIN_USERNAMES` из `.env`, `config`, middleware и документации;
- админы: таблица `admins` + аварийный `SUPERADMIN_IDS` в `.env`;
- все админские действия писать в `audit_logs`.

Критерии приемки:
- пользователь с любым username, но без нужного ID, не получает доступ;
- пользователь с нужным ID проходит, даже если username пуст/изменен.

### REL-04: Корректная проверка прокси
Требования:
- проверять не только TCP-порт, но и реальную работоспособность прокси:
  - handshake/авторизация для SOCKS5/HTTP;
  - тестовый запрос через прокси к контрольному endpoint;
  - опционально тест подключения к Telegram DC;
- статусы прокси: `alive`, `auth_failed`, `timeout`, `dead`;
- хранить `last_error`, `latency_ms`, `last_checked_at`;
- батч-проверка с ограничением concurrency.

Критерии приемки:
- прокси с неверными учетными данными получает `auth_failed`, а не `alive`.

### REL-05: Полная очистка при удалении аккаунта
Требования:
- единый сценарий `hard_delete_account(acc_id)`:
  - отключить активный клиент;
  - удалить `.session`, `.session-wal`, `.session-shm`, `.session-journal`;
  - отвязать прокси;
  - удалить/очистить связки в БД в транзакции;
  - записать действие в `audit_logs`.

Критерии приемки:
- после удаления нет сессионных файлов и активного клиента в памяти.

## 3) Монетизация и доступ

### BIZ-01: Бесплатный тест 24 часа
Требования:
- trial запускается 1 раз на пользователя;
- длительность строго 24 часа с момента активации;
- по окончании trial доступ к рабочим функциям закрывается (paywall).

### BIZ-02: Подписки через ЮKassa
Требования:
- тарифы: 1/3/12 месяцев;
- создание платежа через YooKassa API + `idempotence_key`;
- webhook обработчик `payment.succeeded` / `payment.canceled`;
- идемпотентная обработка webhook;
- автопродление доступа при успешной оплате;
- хранение платежей и статусов в таблице `payments`.

### BIZ-03: Контроль доступа
Требования:
- middleware проверки entitlement перед ключевыми действиями;
- статусы: `trial_active`, `subscription_active`, `expired`, `blocked`;
- уведомления за 3/1 день до окончания подписки.

## 4) Админ-панель (максимальная функциональность)

### ADM-01: Роли и доступ
- роли: `superadmin`, `admin`, `support`, `finance`;
- доступ только по Telegram ID;
- матрица прав по ролям.

### ADM-02: Дашборды
- пользователи: new/trial/paid/expired/blocked;
- финансы: выручка, MRR, ARPPU, churn;
- операционные метрики: ошибки, FloodWait, успешность задач, health воркеров.

### ADM-03: Управление пользователями
- поиск/фильтры;
- блок/разблок;
- выдача/снятие подписки вручную;
- продление/сброс trial (только `superadmin`, с аудитом).

### ADM-04: Управление тарифами и платежами
- CRUD тарифов;
- журнал платежей, фильтры, экспорт CSV;
- ручная сверка и пометка статусов (только `finance`+).

### ADM-05: Промокоды и рефералка
- промокоды: процент/фикс, срок, лимит активаций;
- реферальные коды: бонусные дни за приглашения.

### ADM-06: Поддержка и рассылки
- тикеты пользователей, SLA-статусы;
- сервисные рассылки по сегментам.

### ADM-07: Аудит и безопасность
- `audit_logs` на все админские операции;
- журнал webhook событий;
- защита от дублирующих событий.

## 5) Новая функция: постинг в рекламные чаты по лимитам

### CHAT-01: Режим `promo_chat_posting`
Назначение:
Публиковать рекламные сообщения в заранее разрешенные чаты Telegram с заданной частотой и ограничениями.

Ограничения и безопасность (обязательно):
- только allowlist чатов, добавленных админом;
- на чат: `min_delay`, `max_delay`, `max_posts_per_hour`, `max_posts_per_day`;
- на аккаунт: лимиты в час/сутки, cooldown после ошибок/FloodWait;
- дедупликация: не отправлять одинаковый текст в один и тот же чат в пределах окна `dedup_window_hours`;
- обязательная ротация аккаунтов;
- стоп-лист чатов/аккаунтов при высоком проценте ошибок.

Требования к функционалу:
- отдельный режим кампании `promo_chats`;
- сущность `promo_chats`:
  - `chat_id`, `title`, `username`, `is_active`, `allow_posting`;
- поддержка шаблонов сообщений + spintax + UTM-переменные;
- dry-run режим (без отправки) для проверки;
- журнал отправок по каждому чату с причинами пропусков.

Антибан-логика:
- при `FloodWait`: аккаунт уходит в cooldown до `now + e.value`;
- при `PeerFlood`/ограничениях: аккаунт переводится в `limited`;
- автоматическое снижение частоты при ухудшении deliverability.

Критерии приемки:
- постинг работает только в allowlist чатах;
- лимиты на чат и аккаунт строго соблюдаются;
- при FloodWait нет повторных попыток до завершения cooldown;
- в логах видны `sent/skipped/error` и причина.

## 6) БД и миграции
Добавить таблицы:
- `users` (владелец данных, статус доступа);
- `admins` (telegram_id, role, is_active);
- `plans`;
- `subscriptions`;
- `payments`;
- `promo_codes`, `promo_activations`;
- `referrals`;
- `promo_chats`;
- `audit_logs`;
- `support_tickets`.

Изменения:
- привязать бизнес-сущности к `owner_user_id`;
- добавить индексы:
  - `subscriptions(user_id, status, ends_at)`,
  - `payments(user_id, status, created_at)`,
  - `audit_logs(created_at, admin_id)`,
  - `promo_chats(is_active, allow_posting)`.

## 7) API и инфраструктура
- `POST /webhook/yookassa` (идемпотентный обработчик);
- `GET /health` (бот, БД, scheduler, queue);
- retry/backoff для временных ошибок внешних API;
- централизованный secret management (без хранения секретов в логах).

## 8) Нефункциональные требования
- без логирования OTP, паролей, токенов и секретов;
- транзакционность критичных операций;
- ежедневный backup БД;
- тесты на критичные сценарии;
- мониторинг ошибок и алерты.

## 9) Тест-кейсы приемки (минимум)
1. Zip Slip архив отклоняется.
2. Не-админ по ID не получает доступ в админку.
3. Нерабочий прокси не получает `alive`.
4. `hard_delete_account` полностью чистит сессию и коннекты.
5. Trial истекает ровно через 24 часа.
6. Успешный платеж в ЮKassa активирует подписку.
7. Повторный webhook не создает двойное продление.
8. Режим `promo_chats` соблюдает лимиты и cooldown.

## 10) Этапы внедрения
1. Спринт 1: SEC-01, SEC-03, REL-04, REL-05.
2. Спринт 2: users/subscriptions/paywall/trial 24h.
3. Спринт 3: ЮKassa + webhooks + тарифы.
4. Спринт 4: админ-панель (роли, дашборды, платежи, аудит).
5. Спринт 5: `promo_chat_posting` + антибан + аналитика эффективности.
